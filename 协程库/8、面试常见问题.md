项目主要实现了协程和协程调度器，定时器等。同时，对常见的系统的API，如sleep、socket网络编程，和Sockt IO系类接口API进行了hook，实现非阻塞异步的效果。

Hook:

基于IO协程调度器对**sleep系列、SocketIO系列、fd操作系统系列系统调用进行hook封装实现非阻塞调用的异步。**

**问题2：进程、线程和协程的概念：**

答案：进程

进程是操作系统进行资源分配的基本单位，每个进程都有自己独立内存空间。由于进程比较重量开销的资源多，占独立的内存，所以上下文进程间的切换开销大(栈、寄存器、虚拟内存、文件句柄、打开的文件资源信息等)，但相对安全稳定。

线程：

答案：线程

线程是进程的一个实体，是处理器任务调度和执行的基本单位(线程是cpu调度的基本单位)。线程只拥有一点在运行中必不可少的资源(如程序计数器，一组寄存器和栈)，但是它可以和同属一个进程的其他线程共享该进程所获得的全部资源。

协程：

答案：协程

协程，是一种用户态的轻量级线程，协程的调度完全由用户控制(也就是在用户态执行)。协程拥有自己的寄存器和上下文和栈。协程调度切换时，如果是无栈的情况下将寄存器上下文和栈保存到线程的堆区，在切回来的时候，恢复先前保存的寄存器的上下文和栈，直接操作栈则基本没有内核切换的开销，可以不加锁的访问全局变量，所以上下文的切换非常快。

<font style="color:rgb(51, 51, 51);">协程虽然被称为轻量级线程，但在单线程内，协程并不能并发执行，只能是一个协程结束或yield后，再执行另一个协程，而线程则是可以真正并发执行的。其实这点也好理解，毕竟协程只是以一种花里胡哨的方式去运行一个函数，不管实现得如何巧妙，也不可能在单线程里做到同时运行两个函数，否则还要多线程有何用？</font>

**<font style="color:rgb(51, 51, 51);">问题2：切换上下文进程、线程、协程分别保存了什么上下文</font>**

进程：

cpu上下文——寄存器(要清楚有那些寄存器)；

虚拟内存相关——页表；

资源相关——文件句柄和打开文件资源等信息

线程：

线程切换不需要切换虚拟地址空间；

只需要切换cpu寄存器的上下文和少量资源管理上下文。

协程：

部分cpu寄存器；

如当前调用栈栈基地址、代码的执行位置等，当前的上下文保存到线程的堆区。

### 问题2：多进程、多线程、协程
+ 多进程：  
fork()创建子进程，子进程拷贝父进程地址空间，写时复制，代码段相同，执行任务相同。  
exec 系列函数可以在子进程中加载新的可执行程序，将子进程的代码替换为新程序的代码。这样，子进程将执行与父进程不同的任务。
+ 多线程：  
父进程创建多个线程，每个线程有自己的入口函数，执行不同的任务。多个线程共享父进程的资源。
+ 协程：  
每个协程由自己的入口函数，执行不同的任务。  
协程通常是在单线程中运行的，协程可以在线程中实现切换，开销比线程和进程切换小，可以实现高并发。  
协程经常与多线程一起使用。



### 问题3：对称协程和非对协程的应用场景：
对称协程：

1、复杂状态机实现

+ 对称协程适合实现复杂的状态机，尤其是多个状态之间需要相互跳转的情况。
+ 例如：游戏开发中的角色AI和任务状态机。

2、任务间强耦合的情况

+ 如果多个协程之间需要频繁的相互的调用或传递数据，对称协程更自然。
+ 例如：网络协议栈中状态处理，多层协议需要协作完成数据处理。

非对称协程：

1、事件驱动的异步编程

+ 非对称协程适合异步事件流，比如处理网络请求、文件IO等。
+ 例如：异步框架(python的asyncio、Node.js)或Web服务器的请求处理。

2、分层任务处理

+ 在任务层次结构明确的场景中，非对称协程可以简化设计，因为协程只需要将控制权返回给上层。
+ 例如：网络爬虫中，每个任务爬取一个页面，爬取完后返回调度器。

### 问题4：协程的优缺点：
协程优点：

1、高效的资源利用 

+ 轻量级：协程比线程更轻量。一个线程的创建和上下文切换开销较大，而协程的创建和切换的开销要小的很多。

2、简化异步编程

+  可读性高：使用协程可以编写看起来好像同步代码的异步代码，提升代码的可读性和维护性。

3、非阻塞操作

+ 提高性能：协程允许非阻塞操作，使得程序可以在等待IO或其他耗时操作时继续执行其他任务，提高了程序并发和响应性

4、适用于高并发场景

+ 处理大量并发任务：由于协程开销小，适合处理大量并发任务，如网络请求或高并发的IO操作。

协程缺点：

+ 无法利用多核资源：线程才是系统调度的基本单位，单线程下的多协程本质上还是串行执行的，只能用到单核计算资源，所以协程往往要与多线程、多进程一起使用。 
+ 难以调试：由于协程的切换和异步执行，调试协程代码可能更加困难。当协程之间存在复杂的依赖关系和交互时，追踪问题的根因可能变得复杂。

### 问题5：协程有什么用：
提高资源利用率，提高程序并发性能。协程允许开发者用同步的方式编写异步代码，实现非阻塞的并发操作，通过在适当的时候挂起核恢复协程，可以有效管理多个任务的执行，提高程序的并发性能。

<font style="color:rgb(51, 51, 51);">与线程相比，协程是轻量级的，它们的创建和上下文切换开销较小，可以同时执行大量的协程，而不会导致系统负载过重，可以在单线程下实现异步，使程序不存在阻塞阶段，充分利用cpu资源。简化异步编程逻辑。使用协程可以简化并发编程的复杂性，通过使用适当的协程库或语言特性，可以避免显式的线程同步、锁和互斥量等并发编程的常见问题，用同步的思想就可以编写成异步的程序。</font>

### 问题6：非对称协程使用于I/O密集型任务的原因：
非阻塞IO条件下，等待IO的过程中，会切换到其他任务继续执行；在本项目中通过利用hook加IO协程调度，对SocketIO的API进行，改写使其行为变成非阻塞，意味着我能做recv或sleep函数时，暂停对函数的执行转而去调度其他任务，这样提高了整体的并发性。

### 问题7：协程实现的是真正的异步吗？
<font style="color:rgb(51, 51, 51);">底层使用的是同步非阻塞IO,协程还是同步的，但是可以实现异步的效果，调用返回后，数据就已经读取完成，期间不需要用户干预。</font>

**协程和异步的区别？**

+ 异步是基于事件驱动的非阻塞操作，适用于需要高效管理 I/O 操作的场景，代码较为复杂，容易陷入回调地狱。
+ 协程是轻量级的线程替代品，适用于管理多个任务的高并发场景，代码结构更接近同步方式，避免了回调地狱，并且通过协程的调度机制提高了任务的执行效率。
+ <font style="color:rgb(51, 51, 51);">基于协程的好处就是以同步方式写异步代码，就没有callback地狱了。这时候还是调封装好的API，该API会以非阻塞的方式调用真正的API，如果返回EAGAIN(缺少资源)，那么就往epoll里订阅一个事件，并且通过协程调度器切换到下一个协程，等到条件满足的时候，epoll就会修改协程为可执行状态，等到这个协程被调度执行的时候，再调一次API就可以成功操作资源了。</font>

### Hook的实现细节：
在Hook中如何处理原始函数调用？

使用了宏定义定义了其原始系统调用的类型配合dlsym RTLD_NEXT可以从外挂式的hook中获取原始系统调用的api。这样我们就可以在自己写的同名函数api中调用原始系统调用了。  
Hook是否会影响系统调用性能？

+ **Hook 会影响性能**，但影响的程度取决于系统调用的性质和协程调度的实现方式。如果系统调用涉及 I/O 等阻塞操作，使用 Hook 实现非阻塞行为能够有效提高性能，避免了传统同步 I/O 的阻塞等待。但如果协程的切换过于频繁，或者 Hook 的实现过于复杂，也可能会导致性能下降。
+ **性能提升与开销平衡**：通常来说，使用 Hook 提供非阻塞 I/O 在 I/O 密集型应用中会获得显著性能提升，但在高并发、低延迟要求的场景下，协程的切换开销和 Hook 的附加检查可能会对系统性能产生不小的影响。因此，在设计时需要综合考虑协程的使用场景、系统调用的类型以及任务的性质。

## 作者亲身面试被问过的问题：
### 【滴滴】衡量一个协程库性能的标准：
+ **响应时间：**衡量在给定负载下协程调度的快慢。较低的响应时间表示更高的性能。
+  **吞吐量：**指在单位时间内执⾏的任务数量。较⾼的吞吐量表示更⾼的性能。 
+ **并发能⼒：**指同时处理的协程数量。更⾼的并发能⼒表示更好的性能。 
+ **上下⽂切换开销：**在协程库中，上下⽂切换是指从⼀个协程切换到另⼀个协程的操作。较低的上下⽂切换开销 表示更⾼的性能，因为过多的上下⽂切换会浪费时间和资源。
+ ** 资源利⽤率：**⼀个好的协程库应该有效地利⽤系统资源，例如处理器、内存和⽹络等。协程库在资源利⽤⽅⾯ 的优化程度可以影响性能。

<font style="color:rgb(51, 51, 51);">测试方法：使用原生epoll和本项目的协程库，分别在单线程和多线程条件下，编写服务端程序。服务端接收到请求后，回复一个简单的页面。测试工具：Apache Bench(ab)结论： 单线程情况下，相比于直接使用epoll，性能并无太大差异。多线程情况下，在IO密集、线程或协程切换情况较多时，使用协程有明显的性能优势。</font>

<font style="color:rgb(51, 51, 51);">每个调度线程中，创建一个协程池。</font>

<font style="color:rgb(51, 51, 51);">开始调度之前，先创建一定数量的协程；调度时，选择空闲的协程绑定任务进行调度。如果没有空闲的协程，则创建一个新的协程，可以选择是否添加进入协程池。</font>

<font style="color:rgb(51, 51, 51);">协程池的存在，避免了部分协程的创建和析构，在一定程度上提升了系统的性能。</font>

### 困难2: hook
**问题：**

怎么使用户通过同步编程的方式，实现异步效果？

**解决办法：**

对IO系统调用进行封装，如果fd阻塞，则为fd注册对应的IO事件或定时事件，然后退出当前协程，切换到调度协程；

当事件发生后，触发epoll_wait将事件添加到任务队列中，线程拿取后，协程调度器开始调度执行。这样，用于就使用了简单的添加事件的同步方式，实现了异步效果。

<font style="color:rgb(51, 51, 51);">比如，读事件：首先，用户写一个函数，用于读一个fd，读取之后再进行对应的处理。用户不需要等待读事件完成，如果阻塞，那么就切换到其它任务执行，fd可读时，对应的协程自动会被添加到调度队列，然后开始读，看似我们执行了读事件处理完后才执行其他任务，实际上是可能其他任务先处理完才执行了读事件，但是用户不需要显示编写异步回调的过程，而是利用同步编写的操作完成了异步，这就是协程做的。</font>

**<font style="color:rgb(51, 51, 51);">注：但其本身还是一个同步非阻塞而不是真正的异步。</font>**
